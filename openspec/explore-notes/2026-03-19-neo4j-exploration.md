# SREmate × Neo4j 探索笔记
> 开始日期：2026-03-19，续讨论：2026-03-20

---

## 背景

SREmate 当前本体论引擎现状：
- 10 个实体，9 条关系，全部定义在 `domain-ontology.yaml`
- 引擎做 BFS 路径规划，交给 Gateway 查 MySQL
- LLM 只做意图识别（entity + value + queryScope 三个参数）
- 这是"参数化模板"模式，准确率高但表达能力受限

---

## 已确认的现有限制

1. **只能单向遍历**（从 Order 往下，不能从 Contract 反查 Order）
2. **无条件过滤**（查 ContractNode 就是全部，不能加 status='pending'）
3. **无跨层聚合**（无法统计订单下所有合同的总金额）
4. **路径必须预定义**（YAML 里没有的路径，查不到）
5. **SubOrder 双参数是特例**，破坏了 Gateway 接口统一性

---

## Neo4j 能力对比

### 三种使用姿势

| 姿势 | Neo4j 存什么 | MySQL | 改造成本 | 能力提升 |
|------|-------------|-------|---------|---------|
| A（全量）| 所有业务数据 | 废弃 | 极高 | 最强 |
| B（双存）| 图结构 + 关键字段 | 保留全量数据 | 中 | 强 |
| C（仅Schema）| 实体/关系类型定义 | 保留全量数据 | 低 | 有限 |

**结论**：当前 YAML 本质上就是姿势 C 的 YAML 版本，只存元数据，不存数据。

### Text2Cypher 准确率（GPT-4 级别）
- 单跳简单查询：85–92%
- 双跳：65–75%
- 加 Schema 注入 + Few-shot：可到 80%+
- **生产级高准确率系统倾向于不用 Text2Cypher，而是参数化模板**

## 头部 AI 公司的最佳实践

| 模式 | 代表 | 适合场景 |
|------|------|---------|
| LLM 直接生成 Cypher | Neo4j + LangChain | 查询模式多变 |
| 混合路由（图+向量）| Microsoft GraphRAG、Google | 企业知识库 |
| 参数化模板 | Goldman Sachs、金融医疗 | 高准确率要求 |

**SREmate 现在就是参数化模板模式，这是正确的选择。**

## 核心洞察（待明天继续讨论）

### 洞察 1：最有价值的 Neo4j 引入方式
不是"让 LLM 生成 Cypher"，而是：
> 用 Neo4j 替换 YAML + BFS，做更强的路径规划
> LLM 还是只填参数，准确率不受影响

### 洞察 2：能解锁的新能力（姿势 B）
- 反向遍历：从 Contract 找 Order
- 条件过滤：只查 status=pending 的节点
- 影响范围分析：某个 config 被多少合同引用
- 异常溯源：从 SubOrder 往上追溯到 Order

### 洞察 3：还没讨论的问题
- [ ] 姿势 B 的数据同步方案（CDC vs 定时同步 vs 双写）
- [ ] Neo4j 路径规划 vs 现有 BFS，具体怎么替换
- [ ] SREmate 作为教程项目，这个复杂度值得吗？
- [ ] 有没有更轻量的替代方案（如 Apache AGE、TigerGraph）

## 续讨论（2026-03-20）：LLM 在 SREmate 中的角色定位

### LLM 的实际角色

LLM 在 SREmate 中扮演的是**意图翻译器**，而非查询执行者：

```
用户自然语言
     │
     ▼
   LLM（意图识别）
     │  输出三个参数：
     │  - entity：起始实体（根据编号格式判断）
     │  - value：起始值
     │  - queryScope：目标实体（根据用户意图判断）
     ▼
OntologyQueryEngine（路径规划 + 执行）
     │  BFS 从 entity 找到 queryScope 的路径
     ▼
Gateway（数据查询）
     │  查 MySQL 返回数据
     ▼
DirectOutput（结果输出）
```

LLM 不做路径规划，不生成 SQL/Cypher，只填两个语义参数（起点 + 终点）。这是参数化模板模式，准确率高。
### 核心识别规则

| 识别类型 | 规则 | 示例 |
|---------|------|------|
| entity 选择 | 纯数字 → Order；C 开头 → Contract | 825xxx→Order，C176→Contract |
| queryScope 选择 | 用户说什么数据就传什么实体名 | "合同节点" → ContractNode |

### 讨论点 1：中文名到实体的映射维护在哪里？

**当前问题**：映射规则分散在两处：
- YAML：有 `aliases`，但主要给引擎侧用
- Prompt（`sre-agent.md`）：有决策表，给 LLM 用，人工维护

**两种方案对比**：

| | LLM 直接输出内部名（现状）| 引入抽象层 |
|--|--|--|
| 速度 | 快，无翻译 | 略慢（<1ms 映射） |
| 准确率 | 高（直接训练） | 取决于 YAML alias 质量 |
| 新增实体成本 | 改 YAML + 改 Prompt | 只改 YAML |
| 重构内部名成本 | 改 YAML + 改 Prompt | 只改 YAML |
| 提示词与代码同步风险 | 存在（已出过 bug） | 消除 |
| 实现复杂度 | 低（已完成）| 中等 |

**结论（已达成共识）**：

> **抽象层的本质 = 把「中文名 → 实体」的映射从 Prompt 迁移到 YAML**

当前 YAML 的 `aliases` 和 `{{ontology_summary}}` 注入机制已经具备基础条件，
只需：
1. 充实 YAML 中每个实体的 `aliases`（覆盖用户可能说的中文关键词）
2. 调整 `{{ontology_summary}}` 的注入模板，让 LLM 从注入的实体描述中自行推断 queryScope
3. 移除 Prompt 中的硬编码决策表

**决策**：这是 SREmate v2 的**第一个待优化方向**。

**适合引入的时机**：实体增到 20+ 且 Prompt 维护开始成为真实痛点时；当前 10 实体可先充实 YAML aliases 作为过渡。
