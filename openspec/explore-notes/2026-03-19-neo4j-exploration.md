# SREmate × Neo4j 探索笔记
> 开始日期：2026-03-19，未完成，待续

---

## 背景

SREmate 当前本体论引擎现状：
- 10 个实体，9 条关系，全部定义在 `domain-ontology.yaml`
- 引擎做 BFS 路径规划，交给 Gateway 查 MySQL
- LLM 只做意图识别（entity + value + queryScope 三个参数）
- 这是"参数化模板"模式，准确率高但表达能力受限

---

## 已确认的现有限制

1. **只能单向遍历**（从 Order 往下，不能从 Contract 反查 Order）
2. **无条件过滤**（查 ContractNode 就是全部，不能加 status='pending'）
3. **无跨层聚合**（无法统计订单下所有合同的总金额）
4. **路径必须预定义**（YAML 里没有的路径，查不到）
5. **SubOrder 双参数是特例**，破坏了 Gateway 接口统一性

---

## Neo4j 能力对比

### 三种使用姿势

| 姿势 | Neo4j 存什么 | MySQL | 改造成本 | 能力提升 |
|------|-------------|-------|---------|---------|
| A（全量）| 所有业务数据 | 废弃 | 极高 | 最强 |
| B（双存）| 图结构 + 关键字段 | 保留全量数据 | 中 | 强 |
| C（仅Schema）| 实体/关系类型定义 | 保留全量数据 | 低 | 有限 |

**结论**：当前 YAML 本质上就是姿势 C 的 YAML 版本，只存元数据，不存数据。

### Text2Cypher 准确率（GPT-4 级别）
- 单跳简单查询：85–92%
- 双跳：65–75%
- 加 Schema 注入 + Few-shot：可到 80%+
- **生产级高准确率系统倾向于不用 Text2Cypher，而是参数化模板**

---

## 头部 AI 公司的最佳实践

| 模式 | 代表 | 适合场景 |
|------|------|---------|
| LLM 直接生成 Cypher | Neo4j + LangChain | 查询模式多变 |
| 混合路由（图+向量）| Microsoft GraphRAG、Google | 企业知识库 |
| 参数化模板 | Goldman Sachs、金融医疗 | 高准确率要求 |

**SREmate 现在就是参数化模板模式，这是正确的选择。**

## 核心洞察（待明天继续讨论）

### 洞察 1：最有价值的 Neo4j 引入方式
不是"让 LLM 生成 Cypher"，而是：
> 用 Neo4j 替换 YAML + BFS，做更强的路径规划
> LLM 还是只填参数，准确率不受影响

### 洞察 2：能解锁的新能力（姿势 B）
- 反向遍历：从 Contract 找 Order
- 条件过滤：只查 status=pending 的节点
- 影响范围分析：某个 config 被多少合同引用
- 异常溯源：从 SubOrder 往上追溯到 Order

### 洞察 3：还没讨论的问题
- [ ] 姿势 B 的数据同步方案（CDC vs 定时同步 vs 双写）
- [ ] Neo4j 路径规划 vs 现有 BFS，具体怎么替换
- [ ] SREmate 作为教程项目，这个复杂度值得吗？
- [ ] 有没有更轻量的替代方案（如 Apache AGE、TigerGraph）

## 明天继续的切入点

可以从这几个方向选一个：
1. **具体设计**：姿势 B 的架构怎么落地（Neo4j + MySQL 双存，数据如何同步）
2. **可行性验证**：做一个最小 Demo，把当前 YAML 本体导入 Neo4j，看路径规划效果
3. **价值判断**：作为教程项目，引入 Neo4j 的教学价值是否足够
4. **替代方案**：不用 Neo4j，在现有引擎上做哪些增强最划算
