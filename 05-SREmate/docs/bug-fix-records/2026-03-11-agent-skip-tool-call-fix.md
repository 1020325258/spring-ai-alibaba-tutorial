# Bug 修复记录：Agent 跳过工具调用直接输出数据

**修复日期**：2026-03-11
**修复人**：Claude Code

---

## 问题描述

**用户输入**：`826031111000001859合同数据`

**期望行为**：调用 `queryContractsByOrderId` 工具获取实时数据

**实际行为**：没有工具调用日志，Agent 直接输出了 JSON 数据

```
你: 826031111000001859合同数据

SRE助手:
{"contracts":[{"contractCode":"C1772925352128725","type":8,"status":4,"amount":353.00}]}
⏱ 首字节: 361ms  总耗时: 785ms
```

**关键证据**：没有 `[TOOL]` 日志记录，说明 Agent 没有调用任何工具。

**触发条件**：多轮对话后，Agent 开始跳过工具调用。问题随着对话轮次增加而出现。

---

## 根因分析

**根本原因**：`SREConsole` 维护了 `conversationHistory` 并在每轮请求时传给 LLM。多轮对话后，LLM 从对话历史中"学习"了模式：

1. 用户之前查询过报价单，对话历史中有报价单相关的 JSON 数据格式
2. 当用户再次提到"合同数据"时，LLM 根据"订单号 + 关键词 → JSON 输出"的模式直接生成 JSON
3. LLM 跳过了工具调用环节，"凭记忆"模仿历史数据的格式

**之前的修复为什么不够**：

之前的修复（数据查询结果不写入对话历史）只能防止 LLM 直接复读历史数据，但无法阻止 LLM 学习"输入→输出"的模式并据此生成新数据。

---

## 最终修复方案

**文件**：`src/main/java/com/yycome/sremate/trigger/console/SREConsole.java`

**修改**：完全关闭对话历史功能，每次请求独立处理

```java
// 注：关闭对话历史功能，每次请求独立处理，防止 LLM 模仿历史模式跳过工具调用
// private final List<Message> conversationHistory = new ArrayList<>();
// private static final int MAX_HISTORY = 10;
```

```java
// 每次请求独立处理，不传入对话历史，防止 LLM 模仿历史模式跳过工具调用
Disposable sub = sreAgent.prompt()
        .user(input)
        .stream()
        .content()
```

**删除的代码**：
- `conversationHistory` 字段
- `MAX_HISTORY` 常量
- 历史管理逻辑（添加用户消息、添加助手消息、限制历史长度）
- 相关导入（`AssistantMessage`, `Message`, `UserMessage`, `ArrayList`, `List`）

## 经验总结

### 问题本质

LLM 的"模式学习"能力是一把双刃剑：
- 正面：帮助 LLM 理解用户意图、保持对话连贯
- 负面：LLM 可能根据模式"走捷径"，跳过必要的工具调用

### 为什么必须关闭对话历史

1. **数据查询场景的特殊性**：用户期望每次查询都是实时数据，不需要上下文记忆
2. **提示词约束的局限性**：无论提示词写得多明确，LLM 都可能"忘记"遵守
3. **模式学习的不可控性**：LLM 会从历史中提取模式，这是其核心能力，难以通过规则完全抑制

### 适用场景

**关闭对话历史适用于**：
- 纯数据查询场景（每次查询独立、无上下文需求）
- 需要 100% 保证工具调用的场景
- 不需要多轮对话记忆的场景

**保留对话历史适用于**：
- 需要多轮对话、上下文理解
- 运维诊断类对话（需要追问、澄清）
- 允许一定灵活性的场景

### 设计决策

对于 SREmate，我们选择**完全关闭对话历史**，因为：
1. 核心功能是数据查询，每次都需要实时数据
2. 工具调用是核心能力，必须保证 100% 执行
3. 用户可以一次性描述清楚需求，不需要多轮澄清
