# 合同弹窗诊断

## 触发条件

用户说"合同弹窗提示请先完成报价"、"销售合同发起提示无定软电报价"、"合同发起时弹窗没有数据"、"销售合同/正签合同弹窗异常"

## 两步查询路径

### 第一步：验证用户描述（查询弹窗数据）

从 Order 出发，直接查询 SignableOrderInfo（弹窗可签约S单）：

```
ontologyQuery(entity=Order, value={订单号}, queryScope=SignableOrderInfo)
```

- 可签约S单属于订单维度，Gateway 直接以 `projectOrderId` 调用 `sign-order-list` 接口
- 若弹窗数据为空 → 用户描述属实，进入第二步

### 第二步：查询所有 S 单（仅当弹窗数据为空时）

从 Order 出发，直接查询 SubOrder：

```
ontologyQuery(entity=Order, value={订单号}, queryScope=SubOrder)
```

## 有效 S 单定义

**有效 S 单**：状态码不在 [9001, 9002] 范围内的 S 单

- 9001：已取消
- 9002：已退款
- 其他状态：视为有效 S 单

## 决策矩阵

| 弹窗数据 | S 单情况 | 结论 | 用户描述是否属实 |
|---------|---------|------|----------------|
| **有数据** | - | **用户描述不属实**，系统弹窗数据正常，可签约S单存在 | ❌ 不属实 |
| 空 | 无 S 单 | 用户描述属实，未报价或报价未下单，无可签约S单 | ✅ 属实 |
| 空 | 全部 9001/9002 | 用户描述属实，所有S单已取消/退款，无可签约S单 | ✅ 属实 |
| 空 | 有有效 S 单 | 用户描述属实，弹窗接口逻辑异常，建议人工介入 | ✅ 属实 |

## 排查步骤

1. 根据用户提供的订单号，确定查询起点（Order）
2. 执行第一步：查询弹窗数据（SignableOrderInfo）
   - **若弹窗数据非空 → 用户描述不属实，排查结束**，告知用户系统弹窗数据正常
   - 若弹窗数据为空 → 用户描述属实，继续第二步
3. 执行第二步：查询订单下所有 S 单（SubOrder）
4. 根据决策矩阵输出结论，明确领域归属

## 输出格式

四段式自然语言分析结论：

**【数据查询】** 调用了哪些工具，查到了什么

**【分析】** 根据数据判断用户描述是否属实，引用关键字段值

**【结论】** 用户描述是否属实 + 问题根因（如有）

**【建议】** 下一步操作建议（如有）

## Example

用户: "订单 825123110000002753 的销售合同发起提示无定软电报价"

Action:
1. 查询订单 825123110000002753 下销售合同的可签约S单（SignableOrderInfo）
   - 情形 A：结果非空（有可签约S单）→ **用户描述不属实，排查结束**
     - 结论：系统弹窗数据正常，存在 N 条可签约S单，用户所描述的问题在当前数据中不存在
   - 情形 B：结果为空 → 用户描述属实，继续
2. 查询订单 825123110000002753 下所有 S 单（SubOrder）
   - 结果一：无 S 单 → 用户描述属实，结论：未报价或报价未下单，领域归属报价或订单
   - 结果二：有 S 单但全部状态为 9001/9002 → 用户描述属实，结论：所有S单已取消/退款，领域归属报价或订单
   - 结果三：有有效 S 单 → 用户描述属实，结论：弹窗接口逻辑异常，建议人工介入
